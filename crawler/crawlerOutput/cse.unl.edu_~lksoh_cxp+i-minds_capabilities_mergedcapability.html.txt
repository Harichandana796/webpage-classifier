<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'><o:p> </o:p></span></p>
<p class="MsoNormal" style="margin-bottom:6.0pt"><b style="mso-bidi-font-weight:
normal"><span style='font-size:16.0pt;font-family:"Eras Medium ITC";color:#99CC00'>MergedCapability<o:p></o:p></span></b></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'><o:p> </o:p></span></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'>In version 3, we
decided to merge both the buddy group and question answering capabilities into
one capability.<span style="mso-spacerun:yes">  </span>We refer to these
capabilities as modes within merged capability.<span style="mso-spacerun:yes"> 
</span>There were two good reasons for doing this.<span style="mso-spacerun:yes">  </span>The first is based on the underlying
functionality of capabilities in ConferenceXP.<span style="mso-spacerun:yes"> 
</span>Each capability instance was designed to run autonomously from any other
capabilities being run on the same machine.<span style="mso-spacerun:yes"> 
</span>ConferenceXP provides all the networking between the same capability
being run on multiple machines, but provides no way to communicate between
multiple instances or different capabilities running on the same machine.<span style="mso-spacerun:yes">  </span>This created a good deal of work for us, as
we had to make sure that one or more machines running one or both of our
existing capabilities created no concurrency issues with the database.<span style="mso-spacerun:yes">  </span>The second reason was scalability; suppose we
want to add a new I-MINDS capability or add new tables to the I-MINDS
database.<span style="mso-spacerun:yes">  </span>In the first case we have to
extensively test all three capabilities working in tandem to make sure no
concurrency issues occurred.<span style="mso-spacerun:yes">  </span>In the
second case, we would have to refactor all the database queries separately for
each capability.<span style="mso-spacerun:yes">  </span>This was a time
intensive process and had the potential for introducing new bugs into one or
more I-MINDS capabilities.<o:p></o:p></span></p>
<p class="MsoNormal"><o:p> </o:p></p>
<p class="MsoNormal"><b style="mso-bidi-font-weight:normal"><span style='font-size:16.0pt;font-family:"Eras Medium ITC";color:#FF6600'>1.<span style="mso-spacerun:yes">  </span>Classrooms<o:p></o:p></span></b></p>
<p class="MsoNormal"><b style="mso-bidi-font-weight:normal"><span style='font-family:"Eras Medium ITC"'><o:p> </o:p></span></b></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'>In previous
versions of the I-MINDS capabilities, it was assumed that each group using
I-MINDS would have there own copy of the I-MINDS database.<span style="mso-spacerun:yes">  </span>There was no way for the capabilities to
distinguish between two separate groups using I-MINDS at the same location if
both groups were connected to the same I-MINDS database.<span style="mso-spacerun:yes">  </span>This caused all messages sent by either group
to be lumped together for purposes of generating statistics and natural
language processing.<span style="mso-spacerun:yes">  </span>In version 3 we
have added separate classrooms to the merged capability.<span style="mso-spacerun:yes">  </span>All messages sent in one classroom are
visible only to participants who are also in that classroom.<span style="mso-spacerun:yes">  </span>The classroom is independent of the Venues in
ConferenceXP.<span style="mso-spacerun:yes">  </span>Students in the same
Venue, but different classrooms, will not receive messages.<span style="mso-spacerun:yes">  </span>Modifications to the I-MINDS database allow
messages to be logged and retrieved for specific classrooms.<span style="mso-spacerun:yes">  </span>See section 6 for more information.<o:p></o:p></span></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'><o:p> </o:p></span></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'><o:p> </o:p></span></p>
<p class="MsoNormal"><b style="mso-bidi-font-weight:normal"><span style='font-size:16.0pt;font-family:"Eras Medium ITC";color:#FF6600'>2.<span style="mso-spacerun:yes">  </span>Classrooms<o:p></o:p></span></b></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'><o:p> </o:p></span></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'>In previous
versions of the I-MINDS capabilities, it was assumed that each group using
I-MINDS would have there own copy of the I-MINDS database.<span style="mso-spacerun:yes">  </span>There was no way for the capabilities to
distinguish between two separate groups using I-MINDS at the same location if
both groups were connected to the same I-MINDS database.<span style="mso-spacerun:yes">  </span>This caused all messages sent by either group
to be lumped together for purposes of generating statistics and natural
language processing.<span style="mso-spacerun:yes">  </span>In version 3 we
have added separate classrooms to the merged capability.<span style="mso-spacerun:yes">  </span>All messages sent in one classroom are
visible only to participants who are also in that classroom.<span style="mso-spacerun:yes">  </span>The classroom is independent of the Venues in
ConferenceXP.<span style="mso-spacerun:yes">  </span>Students in the same
Venue, but different classrooms, will not receive messages.<span style="mso-spacerun:yes">  </span>Modifications to the I-MINDS database allow
messages to be logged and retrieved for specific classrooms.<o:p></o:p></span></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'><o:p> </o:p></span></p>
<p class="MsoNormal"><b style="mso-bidi-font-weight:normal"><span style='font-size:16.0pt;font-family:"Eras Medium ITC";color:#FF6600'>3.<span style="mso-spacerun:yes">  </span>Message Object<o:p></o:p></span></b></p>
<p class="MsoNormal"><b style="mso-bidi-font-weight:normal"><span style='font-size:16.0pt;font-family:"Eras Medium ITC";color:#FF6600'><o:p> </o:p></span></b></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'>In version 2 of
the I-MINDS capabilities messages were sent, through the ConferenceXP
networking code, as strings using XML tags to denote the sender, recipient,
etc.<span style="mso-spacerun:yes">  </span>In version 3 we have completed
replaced message strings with a new serialized class called Message.<span style="mso-spacerun:yes">  </span>This class acts as a wrapper for all messages
sent in the merged capability.<span style="mso-spacerun:yes">  </span>This
includes messages sent in buddy group and question answering modes as well as
messages indicating that new participants have joined or left a classroom.<span style="mso-spacerun:yes">  </span>In addition to the message itself, each
Message object contains information about the sender and recipient as well as
an enumeration indicating the message type.<span style="mso-spacerun:yes"> 
</span>Message objects are sent using the ConferenceXP networking code, which
multicasts them to other machines running merged capability.<o:p></o:p></span></p>
<p class="MsoNormal"><b style="mso-bidi-font-weight:normal"><span style='font-size:16.0pt;font-family:"Eras Medium ITC";color:#FF6600'><o:p> </o:p></span></b></p>
<p class="MsoNormal"><b style="mso-bidi-font-weight:normal"><span style='font-size:16.0pt;font-family:"Eras Medium ITC";color:#FF6600'>4.<span style="mso-spacerun:yes">  </span>Merged Capability Agent<o:p></o:p></span></b></p>
<p class="MsoNormal"><b style="mso-bidi-font-weight:normal"><span style='font-size:16.0pt;font-family:"Eras Medium ITC";color:#FF6600'><o:p> </o:p></span></b></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'>In order to
facilitate future agent research, every effort was made to integrate agent
functionality into one base class for version 3.<span style="mso-spacerun:yes">  </span>This Agent class contains three separate
regions controlling most of operations of the merged capability.<span style="mso-spacerun:yes">  </span>The first is the switch region, whenever a
new message is received by the merged capability, the message switch decides
what to do with it based on its message type.<span style="mso-spacerun:yes"> 
</span>For a list of currently supported message types consult table 1.1.<span style="mso-spacerun:yes">  </span>This allows new message types to be added
with relative ease, to support new capability forms, agent communication,
etc.<span style="mso-spacerun:yes">  </span>The second is the message
composition region.<span style="mso-spacerun:yes">  </span>This region contains
methods that are used to create new Message instances in preparation for
sending them through the CXP networking code.<span style="mso-spacerun:yes"> 
</span>The third region in the agent contains methods for running the database
queries.<span style="mso-spacerun:yes">  </span>These methods build the SQL queries
that run select, update and insert operations on the I-MINDS database.<span style="mso-spacerun:yes">  </span><o:p></o:p></span></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'><o:p> </o:p></span></p>
<p align="center" class="MsoNormal" style="text-align:center"><b style="mso-bidi-font-weight:
normal"><span style='font-family:"Eras Medium ITC"'><o:p> </o:p></span></b></p>
<p align="center" class="MsoNormal" style="text-align:center"><b style="mso-bidi-font-weight:
normal"><span style='font-family:"Eras Medium ITC"'><o:p> </o:p></span></b></p>
<p align="center" class="MsoNormal" style="text-align:center"><b style="mso-bidi-font-weight:
normal"><span style='font-family:"Eras Medium ITC"'><o:p> </o:p></span></b></p>
<p align="center" class="MsoNormal" style="text-align:center"><b style="mso-bidi-font-weight:
normal"><span style='font-family:"Eras Medium ITC"'><o:p> </o:p></span></b></p>
<p align="center" class="MsoNormal" style="text-align:center"><b style="mso-bidi-font-weight:
normal"><span style='font-family:"Eras Medium ITC"'>Table 1.1:</span></b><span style='font-family:"Eras Medium ITC"'><span style="mso-spacerun:yes"> 
</span>Message Types Supported<o:p></o:p></span></p>
<p align="center" class="MsoNormal" style="text-align:center"><span style='font-family:"Eras Medium ITC"'><!--[if gte vml 1]><v:shape id="_x0000_i1026"

</v:shape><![endif]--><img border="0" height="171" src="mergedcapability_files/image004.jpg" v:shapes="_x0000_i1026" width="385"/><o:p></o:p></span></p>
<p align="center" class="MsoNormal" style="text-align:center"><span style='font-family:"Eras Medium ITC"'><o:p> </o:p></span></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'>By storing all
the database queries in one class we eliminate the refactoring problems we
encountered in previous versions of I-MINDS.<span style="mso-spacerun:yes"> 
</span>Through the use of one centralized switchboard and methods for creating
Message objects we provide a comprehensive base class for future agent
development.<span style="mso-spacerun:yes">  </span>For example, in order to
add the natural language processing and machine learning from the Q&amp;A capability
we only needed to extend the merged capability agent.<span style="mso-spacerun:yes">  </span>We didn’t even need to modify the switch
code, only overwrite the methods for handling messages from the Q&amp;A
form.<span style="mso-spacerun:yes">  </span>In this sense we regard version 3
of the I-MINDS capability as more than just a CSCL system, but the beginning of
a framework for the same.<o:p></o:p></span></p>
<p class="MsoNormal"><b style="mso-bidi-font-weight:normal"><span style='font-size:16.0pt;font-family:"Eras Medium ITC";color:#FF6600'><o:p> </o:p></span></b></p>
<p class="MsoNormal"><b style="mso-bidi-font-weight:normal"><span style='font-size:16.0pt;font-family:"Eras Medium ITC";color:#FF6600'>5.<span style="mso-spacerun:yes">  </span>Merged Capability Flow of Control<o:p></o:p></span></b></p>
<p class="MsoNormal"><b style="mso-bidi-font-weight:normal"><span style='font-size:16.0pt;font-family:"Eras Medium ITC";color:#FF6600'><o:p> </o:p></span></b></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'>Figure 1.1 shows
the merged capability form in I-MINDS version 3.<span style="mso-spacerun:yes">  </span>In the dropdown box, the user can select
which classroom he or she wants to join.<span style="mso-spacerun:yes"> 
</span>The checkboxes indicate capability modes the user would like to join in
the merged capability.<span style="mso-spacerun:yes">  </span>There are
currently three capability modes:<span style="mso-spacerun:yes">  </span>buddy
group, question answering and structured group.<span style="mso-spacerun:yes"> 
</span>Structured group, however, was disabled for version 3 as it was
implemented but not rigorously tested.<span style="mso-spacerun:yes"> 
</span>For more information on buddy group and question answering modes,
consult sections 2 and 3, respectively.<span style="mso-spacerun:yes"> 
</span>The merged capability maintains separate collections of which modes the
participants have joined.<span style="mso-spacerun:yes">  </span>When a new
participant joins or leaves a capability mode, messages are sent to the other
participants so their collections can be updated.<span style="mso-spacerun:yes">  </span>After the participant presses the join
button, the forms for the capability modes are automatically displayed.<span style="mso-spacerun:yes">  </span>To leave a particular capability mode, the
participant can just close that mode’s form.<span style="mso-spacerun:yes"> 
</span>Closing the merged capability form will cause the participant to leave
all modes.<o:p></o:p></span></p>
