<p class="MsoNormal"><b style="mso-bidi-font-weight:normal"><span style='font-family:"Eras Medium ITC";color:blue'>Data <span class="GramE">Adapters<span style="color:windowtext"><span style="mso-spacerun:yes">  </span></span><span style="color:windowtext;font-weight:normal">One</span></span></span></b><span style='font-family:"Eras Medium ITC"'> significant change we made in this

decision to use the CXP Client, this was a feature that somehow got left
out.<span style="mso-spacerun:yes">  </span>In v2, this feature is back.<span style="mso-spacerun:yes">  </span>The teacher can at any time select the Add Keyword
item from the Options menu.<span style="mso-spacerun:yes">  </span>This brings
up a form containing a <span class="SpellE">DataGrid</span> with all the keywords
and weights stored in the database.<span style="mso-spacerun:yes">  </span>The
user can delete existing keywords by clicking on the row and pressing
delete.<span style="mso-spacerun:yes">  </span>New keywords can be added by
clicking on the star row at the bottom.<span style="mso-spacerun:yes"> 
</span>Existing keywords can be updated by selecting their entries and typing

for teachers.<span style="mso-spacerun:yes">  </span>It displays information

</span>Each piece corresponds to a different statistics category.<span style="mso-spacerun:yes">  </span>The pieces are colored blue if the student is
at or above the thresholds for those statistics (as set by the teacher) and
colored red if the student falls below the set thresholds.<span style="mso-spacerun:yes">  </span>These thresholds are stored in the database
so the teachers can just “set it and forget it”, rather than worry about their
levels.<span style="mso-spacerun:yes">  </span>The statistics are computed for
the last number of sessions specified at the bottom left corner of the
screen.<span style="mso-spacerun:yes">  </span>The Virtual Classroom form is
accessible from the Options menu, in the teacher version.<b style="mso-bidi-font-weight:

thresholds is propagated in both forms.<span style="mso-spacerun:yes">

<p class="MsoNormal"><b style="mso-bidi-font-weight:normal"><span style='font-family:"Eras Medium ITC";color:blue'>Data <span class="GramE">Adapters<span style="color:windowtext"><span style="mso-spacerun:yes">  </span></span><span style="color:windowtext;font-weight:normal">One</span></span></span></b><span style='font-family:"Eras Medium ITC"'> significant change we made in this
version is how the database is accessed by our source code.<span style="mso-spacerun:yes">  </span>In v1 most of our source code other than the
statistics class used C# <span class="SpellE">DataAdapters</span> to access the
database.<span style="mso-spacerun:yes">  </span>In brief, <span class="SpellE">DataAdapters</span>
select all the records in a table and populate a C# <span class="SpellE">DataTable</span>
with the results.<span style="mso-spacerun:yes">  </span>This can be very
convenient because once the <span class="SpellE">DataTable</span> is populated
you can manipulate the data (add, remove, update records) without querying the
database.<span style="mso-spacerun:yes">  </span>However, we encountered
horrible concurrency issues with this implementation.<span style="mso-spacerun:yes">  </span>Because of the way capabilities work, it was
very difficult to synchronize all the <span class="SpellE">DataAdapters</span>.<span style="mso-spacerun:yes">  </span>This led to all sorts of bizarre errors
relating to database concurrency.<span style="mso-spacerun:yes">  </span>The
decision was made to remove all the <span class="SpellE">DataAdapters</span> and
replace them with SQL queries throughout the <span class="SpellE">BuddyGroup</span>
<span class="SpellE">codebase</span>.<span style="mso-spacerun:yes">  </span>This
eliminated the concurrency errors because all the students were dealing
directly with the database.<b style="mso-bidi-font-weight:normal"><o:p></o:p></b></span></p>
<p align="center" class="MsoNormal" style="text-align:center"><span style='font-family:"Eras Medium ITC"'><o:p> </o:p></span></p>
<p class="MsoNormal"><span class="GramE"><b style="mso-bidi-font-weight:normal"><span style='font-family:"Eras Medium ITC";color:blue'>MAX(</span></b></span><span class="SpellE"><b style="mso-bidi-font-weight:normal"><span style='font-family:
"Eras Medium ITC";color:blue'>Session_ID</span></b></span><b style="mso-bidi-font-weight:
normal"><span style='font-family:"Eras Medium ITC";color:blue'>)</span></b><b style="mso-bidi-font-weight:normal"><span style='font-family:"Eras Medium ITC"'><span style="mso-spacerun:yes">  </span></span></b><span style='font-family:"Eras Medium ITC"'>In
v1, the Session ID was determined by simply taking the MAX(<span class="SpellE">Session_ID</span>)
from the database.<span style="mso-spacerun:yes">  </span>This worked well
until we started using Question Answering at the same time.<span style="mso-spacerun:yes">  </span>A scenario occurred where Buddy Group was
started and students joined and queried for their Session ID.<span style="mso-spacerun:yes">  </span>A Question Answering capability was then
started incrementing the Session ID.<span style="mso-spacerun:yes">  </span>If
additional students join the Buddy Group they will no longer receive the
correct Session ID.<b style="mso-bidi-font-weight:normal"><o:p></o:p></b></span></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'><o:p> </o:p></span></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'>In v2, the
Session ID is determined once when the capability is created.<span style="mso-spacerun:yes">  </span>It is then broadcast to each participant when
they join the capability.<span style="mso-spacerun:yes">  </span>In affect, the
Session ID is “buffered” for all participants.<span style="mso-spacerun:yes"> 
</span>Because of this, when the participant who started the capability leaves
the Session ID is not lost.<span style="mso-spacerun:yes">  </span>This also
solves the problem where a student accidentally starts up another Buddy Group
capability.<span style="mso-spacerun:yes">  </span>Now Buddy Group and Question
Answering can coexist peacefully without corrupting the database.<o:p></o:p></span></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'><o:p> </o:p></span></p>
<p class="MsoNormal"><span class="SpellE"><b style="mso-bidi-font-weight:normal"><span style='font-family:"Eras Medium ITC";color:blue'>Students_In_Buddy_Group</span></b></span><b style="mso-bidi-font-weight:normal"><span style='font-family:"Eras Medium ITC";
color:blue'> and <span class="SpellE">Students_In_Session</span></span></b><b style="mso-bidi-font-weight:normal"><span style='font-family:"Eras Medium ITC"'><span style="mso-spacerun:yes">   </span></span></b><span style='font-family:"Eras Medium ITC"'>In
v1, both these tables were N<span class="GramE">:M</span> linking tables
connecting two other tables.<span style="mso-spacerun:yes">  </span>The
limitation on N<span class="GramE">:M</span> linking <span class="SpellE">talbes</span>
is that each pair of values is a unique primary key.<span style="mso-spacerun:yes">  </span>The database will not allow us to have the
same student in the same buddy group twice, for example.<span style="mso-spacerun:yes">  </span>In affect, we knew which students were in
which Buddy Group or Sessions, but that is all we knew.<span style="mso-spacerun:yes">  </span>We wanted to be able to track which students where
in a buddy group during an interval of sessions and when students joined and
left a session.<span style="mso-spacerun:yes">  </span>This required more than
just N<span class="GramE">:M</span> linking tables.<b style="mso-bidi-font-weight:
normal"><span style="mso-spacerun:yes">  </span></b>Both the above tables were
replaced with triply-linked tables.<span style="mso-spacerun:yes">  </span><o:p></o:p></span></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'><o:p> </o:p></span></p>
<p class="MsoNormal"><b style="mso-bidi-font-weight:normal"><span style='font-family:"Eras Medium ITC";color:blue'>Load</span></b><b style="mso-bidi-font-weight:normal"><span style='font-family:"Eras Medium ITC"'>
<span class="GramE"><span style="color:blue">Messages</span><span style="mso-spacerun:yes">  </span><span style="font-weight:normal">In</span></span></span></b><span style='font-family:"Eras Medium ITC"'> v1, each message was broadcast to each
student immediately after it was added to the database.<span style="mso-spacerun:yes">  </span>This worked fine in practice, but happens if
one of your buddies arrives late?<span style="mso-spacerun:yes">  </span>We
decided it would be nice to have the option to ask the database for the
messages sent to your buddies for this session.<span style="mso-spacerun:yes"> 
</span>We added the “Load Messages” menu item to the Options menu.<span style="mso-spacerun:yes">  </span>Pressing it queries the database for the
messages sent during this session.<span style="mso-spacerun:yes">  </span>This
list is filtered by who your buddies are.<span style="mso-spacerun:yes"> 
</span>It will not return messages for students who are not currently your
buddies.<span style="mso-spacerun:yes">  </span>These messages are identically
formatted to those received in real time.<b style="mso-bidi-font-weight:normal"><o:p></o:p></b></span></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'><o:p> </o:p></span></p>
<p class="MsoNormal"><span class="SpellE"><span class="GramE"><b style="mso-bidi-font-weight:
normal"><span style='font-family:"Eras Medium ITC";color:blue'>MessageTree</span></b></span></span><span class="GramE"><b style="mso-bidi-font-weight:normal"><span style='font-family:
"Eras Medium ITC"'><span style="mso-spacerun:yes">  </span></span></b><span style='font-family:"Eras Medium ITC"'>The</span></span><span style='font-family:
"Eras Medium ITC"'> <span class="SpellE">MessageTree</span> class was modified so
that it has the ability to generate a full message tree for any message
regardless of its position in the tree.<span style="mso-spacerun:yes">  
</span>This is done by first determining the root for message tree (the message
ID which started the topic) and then generating the full <span class="SpellE">subtree</span>
from this message.<span style="mso-spacerun:yes">  </span>This reduces the
complexity of some of the queries in the Statistics class considerably.<span style="mso-spacerun:yes">  </span>The <span class="SpellE">MessageTree</span>
class can be configured to only generate the <span class="SpellE">subtree</span>
for the given message ID.<span style="mso-spacerun:yes">  </span><b style="mso-bidi-font-weight:normal"><o:p></o:p></b></span></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'><o:p> </o:p></span></p>
<p class="MsoNormal"><span class="GramE"><b style="mso-bidi-font-weight:normal"><span style='font-family:"Eras Medium ITC";color:blue'>Statistics</span></b><b style="mso-bidi-font-weight:normal"><span style='font-family:"Eras Medium ITC"'><span style="mso-spacerun:yes">  </span></span></b><span style='font-family:"Eras Medium ITC"'>It</span></span><span style='font-family:"Eras Medium ITC"'> was determined through testing that
several of the statistics queries in v1 where not returning exactly the results
we wanted.<span style="mso-spacerun:yes">  </span>A minor change to the way we
stored messages in the database considerably reduced the complexity of some of
the queries while increasing the validity of their results.<span style="mso-spacerun:yes">  </span>This was done by storing whether or not the
message was a topic-starter or response in the Message table.<span style="mso-spacerun:yes">  </span>The Recipient column now contains the value
“BUDDIES-TOPIC” or “BUDDIES-RESPONSE” where appropriate.<span style="mso-spacerun:yes">  </span><b style="mso-bidi-font-weight:normal"><o:p></o:p></b></span></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'><o:p> </o:p></span></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'>The queries in
v1 for Dialog Involvement only looked at the responses and their parents.<span style="mso-spacerun:yes">  </span>Thus they only worked only for message trees
of max height 2.<span style="mso-spacerun:yes">  </span>In v2, we can query for
all the messages that are “BUDDIES-TOPIC” and use the full message tree feature
described above in <span class="SpellE">MessageTree</span> to compute the
statistics for message trees of any height.<span style="mso-spacerun:yes"> 
</span>We were also able to clean up some of the statistics queries in v2 such
as those for <span class="SpellE">ComputeAverageResponseTime</span>.<span style="mso-spacerun:yes">  </span>All queries now check the same session
interval.<span style="mso-spacerun:yes">  </span>In v1, <span class="SpellE">ComputeSessionMessages</span>
and <span class="SpellE">ComputeSessionMessageToTeacher</span> (along with their
averages) did not use this session interval determined by the Calculate
menu.<span style="mso-spacerun:yes">  </span>This has been fixed in v2.<o:p></o:p></span></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'><o:p> </o:p></span></p>
<p class="MsoNormal"><b style="mso-bidi-font-weight:normal"><span style='font-family:"Eras Medium ITC";color:blue'>Database <span class="GramE">Search<span style="color:windowtext"><span style="mso-spacerun:yes">  </span></span><span style="color:windowtext;font-weight:normal">The</span></span></span></b><span style='font-family:"Eras Medium ITC"'> Database Search form can be used by both
students and teachers to look up both messages and users in the database.<span style="mso-spacerun:yes">  </span>Messages are searchable based on the message
itself, the sender, the group they were sent to, by score, by session, and also
by time and date the messages were received, answered, or discarded.<span style="mso-spacerun:yes">  </span>Any combination of these properties can be
used at once thanks to intelligent querying.<span style="mso-spacerun:yes"> 
</span>Students are searchable by name and identifier.<span style="mso-spacerun:yes">  </span>The Database Search form is accessible from
the Options menu.<b style="mso-bidi-font-weight:normal"><o:p></o:p></b></span></p>
