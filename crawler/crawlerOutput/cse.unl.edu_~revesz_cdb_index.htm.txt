<h2 align="center" style="text-align:center"><span style="color:#0070C0;
mso-fareast-language:ZH-CN"> Constraint Databases <o:p></o:p></span></h2>
<p class="MsoNormal"><span style="mso-fareast-language:ZH-CN"> <o:p></o:p></span></p>
<table border="0" cellpadding="0" class="MsoNormalTable" style="mso-cellspacing:1.5pt;
 mso-yfti-tbllook:1184;mso-padding-alt:7.5pt 7.5pt 7.5pt 7.5pt">
<tr style="mso-yfti-irow:0;mso-yfti-firstrow:yes">
<td colspan="2" style="width:733.5pt;padding:7.5pt 7.5pt 7.5pt 7.5pt" valign="top" width="978">
<h4><span style="color:red">Main Idea<o:p></o:p></span></h4>
<p class="MsoNormal"><a name="einleitung">Constraints are natural to describe
  temporal, spatial and geographic data.  Since time and space are
  infinite, spatial and temporal objects are commonly expressed
  over rational or real variables using linear or polynomial
  inequalities.  For example, a rectangle can be described by the
  following constraint </a><span class="SpellE"><span style="mso-bookmark:einleitung">tuple</span></span><span style="mso-bookmark:einleitung">: </span></p>
<p class="MsoNormal"><span style="mso-bookmark:einleitung"> </span></p>
<p align="center" class="MsoNormal" style="text-align:center"><span style="mso-bookmark:einleitung">Rectangle(x, y<span class="GramE">) :</span>- x
  &gt;= 0,  x &lt;= 10,   y &gt;= 0,  y &lt;= 20.</span></p>
<p align="center" class="MsoNormal" style="text-align:center"><span style="mso-bookmark:einleitung"> </span></p>
<p class="MsoNormal"><span style="mso-bookmark:einleitung">Each constraint <span class="SpellE">tuple</span> is a conjunction of inequalities using the
  same set of variables. <span style="mso-spacerun:yes"> </span>In
  the rectangle example the spatial variables are <em>x</em> and <em>y</em>. 
  The Rectangle(x, y) constraint <span class="SpellE"><span class="GramE">tuple</span></span><span class="GramE"><span style="mso-spacerun:yes">  </span>means</span> the set
  of<span style="mso-spacerun:yes">  </span>(x, y) points that satisfy all the
  four inequalities.  There are an infinite number of such points.  A
  constraint relation is a finite set of constraint <span class="SpellE">tuples</span>.
  A constraint database is a finite set of constraint relations.  In 1990,
  <span class="SpellE">Kanellakis</span>, <span class="SpellE">Kuper</span>, and
  Revesz introduced <em>constraint query languages</em> that accept constraint
  databases as input and give constraint databases as output. This <em>constraint
  data model</em> is a natural generalization of the relational data
  model in which relational query languages accept relational databases as
  input and give relational databases as output.  </span></p>
<p class="MsoNormal"><span style="mso-bookmark:einleitung"> </span></p>
<p class="MsoNormal"><span style="mso-bookmark:einleitung">When writing
  constraint database queries, one needs to keep in mind only the schemes of
  the relations and not the actual constraints within the relations. At query
  time the constraint relations can be viewed as only an infinite size
  relation. Hence it is as simple to write constraint database queries as to
  write relational database queries. Nevertheless, one can do more with SQL and
  constraint databases than with SQL and relational databases because the
  infinite size relations are more expressive than finite size relations.<span style="mso-spacerun:yes">  </span>It seems a little magic to be able to do
  more without actually doing more effort in writing queries. However, there is
  some extra work needed inside by the constraint query evaluation engine. In
  general, the constraint query evaluation engine needs to work harder and
  smarter than a relational query evaluation engine. The constraint database
  systems listed below are only a start of implementation efforts. Interested
  in contributing to this area? Learn more about constraint databases from
  the references below and contact us in email.  </span></p>
<p class="MsoNormal"><o:p> </o:p></p>
