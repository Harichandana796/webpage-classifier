<p class="MsoNormal" style="margin-left:24.0pt;text-indent:-24.0pt;mso-list:l6 level2 lfo9;
tab-stops:list 24.0pt"><b style="mso-bidi-font-weight:
normal"><span style='font-family:"Eras Medium ITC";mso-fareast-font-family:
"Eras Medium ITC";mso-bidi-font-family:"Eras Medium ITC";color:red'><span style="mso-list:Ignore">1.1.<span style='font:7.0pt "Times New Roman"'> </span></span></span></b><b style="mso-bidi-font-weight:normal"><span style='font-family:"Eras Medium ITC";
color:red'>Introduction<br style="mso-special-character:line-break"/>
<br style="mso-special-character:line-break"/>
<o:p></o:p></span></b></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'>IMINDS
Capability (CXP+I-MINDS) is an online collaboration system intelligently
supported by a multiagent system integrated into the <span class="SpellE">ConferenceXP</span>
framework designed by Microsoft Research.<span style="mso-spacerun:yes"> 
</span>It can be used to support groups of employees in corporate environments,
as well as students in educational settings by serving as a computer supported,
collaborative learning (CSCL) system designed to promote learning through group
activities. <o:p></o:p></span></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'><o:p> </o:p></span></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'>In version 1, we
began by creating two simple applications:<span style="mso-spacerun:yes"> 
</span><span class="SpellE">BuddyGroup</span>, which allowed for synchronous
communications between groups of users, and <span class="SpellE">QuestionAnswering</span>,
which allowed users to ask questions of a moderator, who could then respond
with an appropriate answer.<span style="mso-spacerun:yes">  </span>In <span class="SpellE">QuestionAnswering</span>, all questions were parsed for important
keywords and were automatically scored using a machine learning algorithm to
assist the moderator in knowing which questions were most important.<span style="mso-spacerun:yes">  </span><span class="SpellE">BuddyGroup</span> also
provided a form for viewing performance statistics tracked for all members of
the user’s group.<span style="mso-spacerun:yes">  </span>All messages passed
between users were automatically tracked and stored in a <span class="SpellE">MySQL</span>
database for archival purposes.<span style="mso-spacerun:yes">  </span><o:p></o:p></span></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'><o:p> </o:p></span></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'>In version 2, we
added several tools for further utilizing the information stored in our
database.<span style="mso-spacerun:yes">  </span><span class="SpellE">DatabaseSearch</span>
was created to allow users to search through archived data and find previously
stored messages based on any combination of metadata stored about the
messages.<span style="mso-spacerun:yes">  </span>It also allowed users to find
other users of the system, and learn how active they were.<span style="mso-spacerun:yes">  </span><span class="SpellE">VirtualClassroom</span>
collected statistics about every user and provided an overview of current
performance for moderators.<span style="mso-spacerun:yes">  </span>Users with
problematic areas (as determined by thresholds set by the moderator) were
flagged for future observation.<span style="mso-spacerun:yes">  </span>To
further analyze each statistic for every user, moderators could use <span class="SpellE">StatCharts</span> to view bar charts of every statistic for a
particular user, or one statistic for every user.<o:p></o:p></span></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'><o:p> </o:p></span></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'>In version 3,
our goal was to combine all of our features into one capability.<span style="mso-spacerun:yes">  </span>This served two purposes:<span style="mso-spacerun:yes">  </span>1) it made things easier for users, who now
had all functionality in one place, and 2) it made things easier for underlying
user and moderator agents because they could now access all available
information about user behavior and actions.<span style="mso-spacerun:yes"> 
</span>In previous versions, all intelligent functionality was contained in the
applications themselves, but 3.0 allowed us to merge all agent functionality
into one object.<span style="mso-spacerun:yes">  </span>We also redesigned our
database for version 3, redoing the hierarchy of tracking.<span style="mso-spacerun:yes">  </span>Before, all events recorded in CXP+I-MINDS
were associated with a specific session (i.e., a discrete instance of system
usage).<span style="mso-spacerun:yes">  </span>In 3.0 and beyond, sessions were
now tied to specific classrooms, which allowed us to further separate data
between different classes using the same database.<span style="mso-spacerun:yes">  </span>Finally, for 3.0, we also revised the CXP
networking stack (in CXP version 3.5) to use the Pragmatic General Multicast
(PGM) protocol to test reliability and latency in wireless environments.<o:p></o:p></span></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'><o:p> </o:p></span></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'>In version 4, we
had several primary goals: 1) build a system robust enough to operate in both
wired and wireless environments,<span style="mso-spacerun:yes">  </span>2)
extend the existing functionalities to include 2 new modes – <span class="SpellE">StructuredChat</span> and <span class="SpellE">QuizMode</span>, 3)
reorganize the agent functionalities into specialties which are components that
specialize in one area of tasks, 4) create an XML configuration file to give
users more flexibility in setting options, 5) add network tracking for PGM and
wireless environments, and 6) redo the main CXP+I-MINDS form to offer more
functionality and be more user friendly.<span style="mso-spacerun:yes"> 
</span>All coding for 4.0 was moved to Visual Studio 2005 to use .NET 2.0 and
be compliant with CXP version 4.0.<span style="mso-spacerun:yes"> 
</span>Statistic tracking was also redone to provide less reliance on database
queries and faster performance, and the statistical analysis tools were redone
to offer better user interfaces and to take advantage of the new tracking
mechanisms.<span style="mso-spacerun:yes">  </span>Finally, <span class="SpellE">BuddyGroup</span>
was renamed to <span class="SpellE">BuddyChat</span> to avoid confusion when
talking about the application and groups of buddies.<b style="mso-bidi-font-weight:
normal"><br style="mso-special-character:line-break"/>
<br style="mso-special-character:line-break"/>
</b><o:p></o:p></span></p>
<p class="MsoNormal" style="margin-left:24.0pt;text-indent:-24.0pt;mso-list:l6 level2 lfo9;
tab-stops:list 24.0pt"><b style="mso-bidi-font-weight:
normal"><span style='font-family:"Eras Medium ITC";mso-fareast-font-family:
"Eras Medium ITC";mso-bidi-font-family:"Eras Medium ITC";color:red'><span style="mso-list:Ignore">1.2.<span style='font:7.0pt "Times New Roman"'> </span></span></span></b><b style="mso-bidi-font-weight:normal"><span style='font-family:"Eras Medium ITC";
color:red'>Wireless Issues and Logger Version<o:p></o:p></span></b></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'><o:p> </o:p></span></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'>After a brief
deployment of version 3 in a wireless classroom environment, we quickly
discovered several flaws in our overall design.<span style="mso-spacerun:yes"> 
</span>One major flaw was our handling of database connections.<span style="mso-spacerun:yes">  </span>In version 3, every object that needed
database access was given its own database connection for running queries.<span style="mso-spacerun:yes">  </span>However, the database connections use TCP/IP
for their data transmissions, so in a classroom of over 30 students, where each
student has multiple connections, this resulted in too much concurrent TCP
traffic which slowed down the user’s connections due to loss and throughput
reductions and caused serious usability issues with 3.0.<span style="mso-spacerun:yes">  </span>This was never a problem over wired
connections because they suffer from far less loss than wireless connections,
so slowdown was never previously an issue.<o:p></o:p></span></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'><o:p> </o:p></span></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'>To counter this
problem, we built 4.0 with reduction in mind – how can we reduce the number of
database connections to ensure faster usability?<span style="mso-spacerun:yes">  </span>How can we reduce the overall amount of
traffic being sent to protect against loss and need for retransmissions through
loss recovery?<span style="mso-spacerun:yes">  </span>Reducing the impact of
CXP+I-MINDS on the network infrastructure it resides in allows our system to be
more robust and scalable, which are two important properties for a good
computer support collaborative learning (CSCL) application to have.<span style="mso-spacerun:yes">  </span>To achieve our goal, we designed a new
paradigm for database access.<span style="mso-spacerun:yes">  </span>First of
all, we eliminated all database connections from the software except for one
per user, which is shared by all of the objects within I-MINDS.<span style="mso-spacerun:yes">  </span>We further added locks to protect against
multiple threads accessing the connection at once and implemented a stronger
error-prevention system using more try-catch handling of exceptions and error
logging.<span style="mso-spacerun:yes">  </span>We also recognized that even 1
connection per user might be too many in wireless environments, so we designed
a logger system, where one instance of CXP+I-MINDS acts as a database mediator
for all participants.<span style="mso-spacerun:yes">  </span>Whenever a user’s
software or agent needs to run a query, it simply sends a message to the
logger, which runs the query and returns the results.<span style="mso-spacerun:yes">  </span>However, such a system introduces extra
messages into the system (especially because I-MINDS relies heavily on its
database to collect and store important information about users’ behavior and
actions), so we had to carefully design our message passing and refine our old
procedures to prevent the system from becoming congested with messages.<span style="mso-spacerun:yes">  </span>Also, such a system is not necessary in wired
environments, so whether or not to use a local database connection can be set
in our new I-MINDS XML configuration file (described in section 1.5).<o:p></o:p></span></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'><o:p> </o:p></span></p>
<p class="MsoNormal"><b style="mso-bidi-font-weight:normal"><span style='font-family:"Eras Medium ITC";color:red'>1.3<span class="GramE">.<span style="mso-spacerun:yes">  </span>New</span> Modes</span></b><span style='font-family:"Eras Medium ITC";color:red'><o:p></o:p></span></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'><o:p> </o:p></span></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'>To further
facilitate collaboration and learning amongst users utilizing CXP+I-MINDS, we
have introduced two new modes:<span style="mso-spacerun:yes">  </span>1) <span class="SpellE">StructuredChat</span>, and 2) <span class="SpellE">QuizMode</span>.<span style="mso-spacerun:yes">  </span><span class="SpellE">StructuredChat</span> is
just like <span class="SpellE">BuddyChat</span>, allowing students to carry on
chat-based discussions about multiple topics.<span style="mso-spacerun:yes"> 
</span>The main difference between the two modes is that <span class="SpellE">BuddyChat</span>
allows users to select their own groups, which <span class="SpellE">StructuredChat</span>
allows moderators to choose the groups for the students.<span style="mso-spacerun:yes">  </span><span class="SpellE">StructuredChat</span> is
described more in detail in Section 3.<o:p></o:p></span></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'><o:p> </o:p></span></p>
<p class="MsoNormal"><span class="SpellE"><span style='font-family:"Eras Medium ITC"'>QuizMode</span></span><span style='font-family:"Eras Medium ITC"'> is primarily for use in educational or
training settings and allows moderators (teachers) to create questions and
quizzes to give to users (students) to assess their understanding of concepts
taught.<span style="mso-spacerun:yes">  </span>Quizzes can either be given to
every student to be taken individually, or they can be given to structured
groups to work on collaboratively.<span style="mso-spacerun:yes">  </span><span class="SpellE">QuizMode</span> is described further in Section 4.<o:p></o:p></span></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'><o:p> </o:p></span></p>
<p class="MsoNormal"><b style="mso-bidi-font-weight:normal"><span style='font-family:"Eras Medium ITC";color:red'>1.4<span class="GramE">.<span style="mso-spacerun:yes">  </span>Agent</span> Specialties<o:p></o:p></span></b></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'><o:p> </o:p></span></p>
<p class="MsoNormal"><span class="GramE"><span style='font-family:"Eras Medium ITC"'>As
the functionalities and abilities of CXP+I-MINDS grow, so does the underlying
user and moderator agents.</span></span><span style='font-family:"Eras Medium ITC"'><span style="mso-spacerun:yes">  </span>To facilitate future changes and maintain
current features, we have moved much of the agent functionalities out of its
main class and into specialties designed to work with specific tasks.<span style="mso-spacerun:yes">  </span>For example, <span class="SpellE">BuddyChat</span>
and <span class="SpellE">StructuredChat</span> each have their own specialties
which contain all of the intelligence necessary for generating statistical
information about user behavior, notifying users of current conditions (such as
many unread messages in a topic or low contribution levels), and finding new
buddies based on topic discussions (in <span class="SpellE">BuddyChat</span>
only).<span style="mso-spacerun:yes">  </span>A modeling specialty was also
created to handle all modeling tasks, such as tracking statistical information
from the various modes and for tracking the current network conditions (if
applicable).<span style="mso-spacerun:yes">  </span>This allows us to build
statistical information incrementally, instead of relying on database queries
to generate values on demand (as was done in previous versions of CXP+I-MINDS),
greatly speeding up the analysis tools described in Section 5.<span style="mso-spacerun:yes">  </span>Finally, a logger specialty was designed to
handle all database access for users and for archiving message content.<o:p></o:p></span></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'><o:p> </o:p></span></p>
<p class="MsoNormal"><b style="mso-bidi-font-weight:normal"><span style='font-family:"Eras Medium ITC";color:red'>1.5<span class="GramE">.<span style="mso-spacerun:yes">  </span>Configuration</span> File<o:p></o:p></span></b></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'><o:p> </o:p></span></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'>In previous
versions of CXP+I-MINDS, important information regarding the behavior of the
application was stored in various places.<span style="mso-spacerun:yes"> 
</span>For example, database connection information was stored in system <span class="SpellE">DSNs</span> in Windows.<span style="mso-spacerun:yes"> 
</span>However, during one deployment, we discovered how difficult this design
can be because users tend to forget to set the proper settings, and <span class="SpellE">DSNs</span> are not easy to find unless you are already familiar
with them.<span style="mso-spacerun:yes">  </span>To make the system more
usable and prevent errors, we moved all configuration information into one XML
file which is read upon starting CXP+I-MINDS.<span style="mso-spacerun:yes"> 
</span>Within this configuration file is the database connection (with
encrypted password for security), whether or not we want to use our own
database connection or rely on a logger, and whether or not to enter debug mode
for extra error messages.<span style="mso-spacerun:yes">  </span>For PGM within
the CXP networking stack, we also created a similar configuration system which
allows users to set all of <span class="SpellE">PGM’s</span> options (whether or
not to use late join, FEC, etc.) in one place.<span style="mso-spacerun:yes"> 
</span>This file also allows different settings for each socket using PGM.<span style="mso-spacerun:yes">  </span>When we ported our PGM code to CXP 4.0, we
retained the UDP protocol, and the networking configuration file also allows
user to choose which protocol they would like to employ.<o:p></o:p></span></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'><o:p> </o:p></span></p>
<p class="MsoNormal"><b style="mso-bidi-font-weight:normal"><span style='font-family:"Eras Medium ITC";color:red'>1.6<span class="GramE">.<span style="mso-spacerun:yes">  </span>Network</span> Tracking<o:p></o:p></span></b></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'><o:p> </o:p></span></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'>One of our goals
as a research team is to provide an online collaboration system robust in
multiple networking environments.<span style="mso-spacerun:yes">  </span>To
achieve this goal, we have been experimenting with the PGM networking protocol
in wireless environments.<span style="mso-spacerun:yes">  </span>Unlike UDP
(which is often chosen for wireless networking), PGM provides reliability
through loss detection and recovery while maintaining high throughput (unlike
traditional TCP connections).<span style="mso-spacerun:yes">  </span>In order
to get a better understanding of what is going on at the network level, we have
added network tracking to version 4 of I-MINDS.<span style="mso-spacerun:yes"> 
</span>Network tracking occurs at 3 levels: 1) the device level, 2) the socket
level, and 3) the application level.<span style="mso-spacerun:yes">  </span>At
the device level, if users are using a wireless card installed with a special
NDIS protocol driver (which we can provide to users), the modeling specialty
can gather information about the network connection including which network and
access point (AP) the user is connected to, the signal strength, and the number
of available <span class="SpellE">APs</span>.<span style="mso-spacerun:yes"> 
</span>At the socket level, if the users are using PGM, we can get valuable
statistical information from the operating system, including how many packets
have been lost, how many have been recovered, what the current transmission
rate is, etc.<span style="mso-spacerun:yes">  </span>These statistics are
available from both the sender and receiver perspective.<span style="mso-spacerun:yes">  </span>Finally, at the application level, the user
agents occasionally pass special messages between one another to determine the
current reliability and latency in the network.<span style="mso-spacerun:yes"> 
</span><o:p></o:p></span></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'><o:p> </o:p></span></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'>Right now,
network tracking is only performed for the purposes of our research for testing
network stability.<span style="mso-spacerun:yes">  </span>However, we are
working on adding functionality to the agents to allow them to use information
about the current network conditions to adjust their transmission strategies to
alleviate pressure on the network and improve overall performance. <o:p></o:p></span></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'><o:p> </o:p></span></p>
<p class="MsoNormal"><b style="mso-bidi-font-weight:normal"><span style='font-family:"Eras Medium ITC";color:red'>1.7<span class="GramE">.<span style="mso-spacerun:yes">  </span>Main</span> Form<o:p></o:p></span></b></p>

the main form to provide easy access to each mode and tool.<span style="mso-spacerun:yes">  </span>We have also improved the functionality for
joining and leaving classrooms.<span style="mso-spacerun:yes">  </span>Finally,
we have added an online user list to show users and moderators who are
currently in the same classroom and which modes they are using.<span style="mso-spacerun:yes">  </span>This same list also allows participants to
invite others to use the different modes.<span style="mso-spacerun:yes"> 
</span>For users, it shows whether or not online participants are in their
buddy or structured groups, as shown in Figure 1.1.<o:p></o:p></span></p>

<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'>To assist both
users and moderators analyze the data tracked by CXP+I-MINDS, we have designed
several tools:<span style="mso-spacerun:yes">  </span>1) <span class="SpellE">VirtualClassroom</span>,

4.<span style="mso-spacerun:yes">  </span>In versions 2 and 3, <span class="SpellE">VirtualClassroom</span> had a limited set of statistics, and also
included an overview of all message traffic from the system in the form of
message and topic lists, but we removed these because they were not used during
deployment.<span style="mso-spacerun:yes">  </span>Also, for version 4, we
included the Statistics tool in the user version to replace its text-based
predecessor, which was available in all prior versions of CXP+I-MINDS, to accommodate
the increase in statistical tracking and to provide information in an easier to
analyze format.<span style="mso-spacerun:yes">  </span><span class="SpellE">DatabaseSearch</span>
has remained unchanged from its design in version 3.<o:p></o:p></span></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'><o:p> </o:p></span></p>
<p class="MsoNormal"><span style='font-family:"Eras Medium ITC"'>All statistical
modeling is done by each user agent based on their user’s interactions with
